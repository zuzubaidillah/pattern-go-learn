Untuk konsep **migration DB yang banyak koneksi**, pola yang paling aman adalah:

**anggap setiap database / role koneksi sebagai target migrasi yang terpisah**, lalu jalankan migrasi **per target**, bukan satu file migration yang bebas loncat-loncat antar koneksi. Tools Go yang umum seperti `golang-migrate` dan `goose` memang dirancang untuk menjalankan migrasi terhadap **satu database target per instance/provider**, jadi untuk multi-DB Anda biasanya membuat **beberapa runner/provider**, masing-masing mengarah ke koneksi yang berbeda. ([GitHub][1])

## Jawaban praktis

Kalau aplikasi Anda punya:

* MySQL primary
* MySQL read replica
* PostgreSQL analytics
* Redis

maka aturan migrasinya:

* **migrasi schema hanya dijalankan ke database write/primary**
* **read replica tidak dimigrasi langsung**
* replica akan ikut berubah lewat replication dari primary
* **setiap engine/database logical yang berbeda** punya folder migrasi sendiri

Jadi misalnya:

```bash id="l9jzv0"
migrations/
├── mysql_main/
│   ├── 000001_create_users_table.up.sql
│   ├── 000001_create_users_table.down.sql
│   └── 000002_add_user_status_index.up.sql
├── postgres_analytics/
│   ├── 000001_create_event_logs.up.sql
│   └── 000001_create_event_logs.down.sql
└── shared_docs/
```

Itu lebih aman daripada satu folder berisi SQL campuran untuk banyak koneksi. `golang-migrate` menjalankan migrasi terhadap satu source dan satu database target per instance, dan `goose` provider juga dibuat dengan `db *sql.DB` tertentu, jadi pola multi-target yang benar adalah membuat runner/provider terpisah untuk tiap DB. ([GitHub][1])

# Prinsip desain yang saya sarankan

## 1. Pisahkan migrasi berdasarkan **database owner**

Jangan berdasarkan module dulu, tapi berdasarkan **database fisik / logical target**.

Contoh bagus:

```bash id="xq7261"
migrations/
├── mysql_core/
├── mysql_billing/
└── postgres_reporting/
```

Kalau semua module Anda masih pakai satu MySQL primary, cukup:

```bash id="z6fobn"
migrations/
└── mysql_main/
```

## 2. Read replica tidak punya folder migrasi sendiri

Karena schema source of truth tetap di primary/write DB.

Jadi ini salah pola:

```bash id="g6v25e"
migrations/
├── mysql_write/
└── mysql_read/
```

Kalau `mysql_read` hanya replica, tidak perlu.

## 3. Satu migration = satu database target

Hindari file seperti:

```sql
USE db_a;
CREATE TABLE ...;

USE db_b;
ALTER TABLE ...;
```

Karena ini membuat:

* rollback sulit
* tracking version kacau
* coupling tinggi
* rawan gagal setengah jalan

Ini juga selaras dengan filosofi `golang-migrate` yang sengaja menjaga driver tetap “dumb” dan fail fast, bukan mengoreksi orkestrasi multi-database secara otomatis. ([GitHub][1])

## 4. Jangan andalkan GORM AutoMigrate untuk migration production penuh

GORM punya `Migrator` dan `AutoMigrate`, tapi dokumentasinya jelas: auto migration fokus pada create table, missing columns, missing indexes, dan historically tidak dimaksudkan sebagai pengganti migration versioned yang ketat untuk perubahan schema kompleks. ([GORM][2])

Untuk production:

* schema migration → pakai `golang-migrate` atau `goose`
* GORM `AutoMigrate` → kalau mau, hanya untuk prototyping/dev awal

# Struktur project yang saya sarankan

Untuk tree Anda, tambahkan seperti ini:

```bash id="u4b3hn"
.
├── cmd
│   ├── api
│   │   └── main.go
│   └── migrate
│       └── main.go
├── migrations
│   ├── mysql_main
│   │   ├── 000001_create_users_table.up.sql
│   │   ├── 000001_create_users_table.down.sql
│   │   └── 000002_add_email_unique_index.up.sql
│   └── postgres_reporting
│       ├── 000001_create_event_logs.up.sql
│       └── 000001_create_event_logs.down.sql
└── internal
    ├── infra
    │   └── db
    │       ├── manager.go
    │       └── migrate.go
```

Kenapa saya sarankan `cmd/migrate/main.go`?

Karena migration sebaiknya jadi **entrypoint tersendiri**, bukan otomatis selalu jalan dari `cmd/api/main.go`.

Ini penting supaya:

* deploy lebih aman
* API pod tidak balapan menjalankan migration
* migrasi bisa dieksekusi satu kali oleh CI/CD/job khusus

Pada `goose`, ada diskusi dan dokumentasi yang menunjukkan migrasi sebaiknya dianggap berjalan secara singleton/sekuensial; menjalankan paralel dari banyak instance bisa menimbulkan race atau kegagalan. ([GitHub][3])

# Flow migration yang aman

## Opsi terbaik: migrasi dijalankan terpisah dari API startup

Urutan deploy:

```text id="fa8mty"
1. build artifact
2. jalankan migration job
3. jika sukses, deploy API baru
4. API instance baru start
```

Ini lebih aman daripada:

```text id="i08v4o"
setiap pod API start → otomatis run migration
```

Karena kalau 3 pod start bersamaan, Anda bisa kena race.

# Contoh konsep multi-connection runner

Misal Anda punya DB manager seperti ini:

```go id="xbj3ik"
// File: internal/infra/db/manager.go
package db

import "database/sql"

type Manager struct {
	MySQLWrite    *sql.DB
	MySQLRead     *sql.DB
	PostgresWrite *sql.DB
	PostgresRead  *sql.DB
}
```

Lalu buat deklarasi target migrasi:

```go id="vztezx"
// File: internal/infra/db/migrate.go
package db

type MigrationTarget struct {
	Name string
	DSN  string
	Dir  string
	Kind string // mysql, postgres
}
```

Contoh resolver target:

```go id="2t4uy3"
// File: internal/infra/db/migrate.go
package db

import "yourapp/internal/config"

func BuildMigrationTargets(cfg config.Config) []MigrationTarget {
	var targets []MigrationTarget

	if cfg.DB.MySQL.Write.DSN != "" {
		targets = append(targets, MigrationTarget{
			Name: "mysql_main",
			DSN:  cfg.DB.MySQL.Write.DSN,
			Dir:  "migrations/mysql_main",
			Kind: "mysql",
		})
	}

	if cfg.DB.Postgres.Write.DSN != "" {
		targets = append(targets, MigrationTarget{
			Name: "postgres_reporting",
			DSN:  cfg.DB.Postgres.Write.DSN,
			Dir:  "migrations/postgres_reporting",
			Kind: "postgres",
		})
	}

	return targets
}
```

Lalu CLI migrate menerima argumen target:

```bash id="w8rn5e"
go run ./cmd/migrate -target=mysql_main up
go run ./cmd/migrate -target=postgres_reporting up
go run ./cmd/migrate -target=all up
```

# Pilih tool apa?

## Kalau ingin simple dan populer: `golang-migrate`

Cocok kalau Anda suka:

* SQL file versioned
* CLI kuat
* deterministic
* jelas untuk CI/CD

`golang-migrate` memang membaca migration dari source dan menjalankannya terhadap database target tertentu. ([GitHub][1])

## Kalau ingin embed dan provider lebih fleksibel: `goose`

`goose` mendukung SQL migration dan Go migration, dan provider-nya bisa dibuat lebih dari satu karena tidak punya global state. Itu bagus untuk multi-target runner di aplikasi Go Anda. ([GitHub][4])

### Rekomendasi saya

Untuk arsitektur Anda, saya lebih condong ke:

* **`golang-migrate`** kalau fokus SQL-only, CI/CD-friendly
* **`goose`** kalau ingin embed migration runner di binary Go dengan fleksibilitas beberapa target

# Konsep version table

Setiap database target harus punya **table version migration sendiri** di databasenya masing-masing.

Artinya:

* MySQL main punya riwayat versi sendiri
* PostgreSQL reporting punya riwayat versi sendiri

Jangan pakai satu version table untuk mengontrol banyak database sekaligus.

Kenapa?

* rollback jadi tidak jelas
* satu target bisa sukses, target lain gagal
* audit sulit

Tool migration memang pada dasarnya melacak status per database target, bukan lintas banyak DB sekaligus sebagai satu unit global. ([GitHub][1])

# Apakah boleh migration per module?

Boleh, tapi saya sarankan **module tetap tunduk ke database target**.

Contoh kompromi yang bagus:

```bash id="cg6vut"
migrations/
├── mysql_main
│   ├── user
│   │   ├── 000001_create_users_table.up.sql
│   │   └── 000001_create_users_table.down.sql
│   ├── auth
│   │   ├── 000002_create_refresh_sessions.up.sql
│   │   └── 000002_create_refresh_sessions.down.sql
│   └── billing
│       ├── 000003_create_invoices.up.sql
│       └── 000003_create_invoices.down.sql
└── postgres_reporting
    └── analytics
```

Tapi perlu hati-hati: beberapa tool/runner lebih sederhana kalau folder flat. Pada `golang-migrate`, pola paling umum adalah satu source URL/folder per target, jadi kalau Anda ingin subfolder modular biasanya perlu orkestrasi runner sendiri atau konsolidasi build step. ([GitHub][1])

Jadi secara praktis, untuk awal lebih aman:

```bash id="tn39s1"
migrations/mysql_main/
  000001_...
  000002_...
  000003_...
```

lalu nama file diberi prefix module:

```text id="12g3kk"
000001_user_create_users_table.up.sql
000002_auth_create_refresh_sessions.up.sql
000003_billing_create_invoices.up.sql
```

# Kalau satu fitur menyentuh dua database?

Contoh:

* create user di MySQL main
* create analytics table/index di PostgreSQL reporting

**Jangan jadikan itu satu migration atomic.**
Perlakukan sebagai dua migration terpisah:

* migration MySQL
* migration PostgreSQL

Lalu orkestrasi deploy Anda menentukan urutan.

Karena secara umum Anda **tidak punya transaksi schema lintas dua database engine berbeda**. Jadi lebih baik eksplisit.

# Best practice yang saya sarankan

## 1. Migration hanya ke primary/write DB

Replica tidak dimigrasi langsung.

## 2. Satu target DB = satu jalur migration

Pisahkan folder dan version history.

## 3. Jalankan migration dari job terpisah

Jangan setiap API instance otomatis migrate.

## 4. Hindari AutoMigrate untuk production changes

Gunakan versioned SQL migration.

## 5. Buat migration backward-compatible untuk zero-downtime

Contoh:

* deploy 1: tambah kolom nullable
* aplikasi baru mulai menulis kolom itu
* deploy 2: backfill
* deploy 3: baru enforce not null / drop kolom lama

## 6. Jangan campur data migration berat dengan schema migration besar

Kalau backfill jutaan row:

* schema migration tetap ringan
* backfill dijalankan job terpisah

## 7. Beri naming target yang jelas

Contoh:

* `mysql_main`
* `postgres_reporting`
* `mysql_tenant_meta`

Jangan nama ambigu seperti `db1`, `db2`.

# Rekomendasi final untuk kasus Anda

Karena Anda punya konsep:

* modular monolith
* multi DB
* read/write separation
* Redis
* high concurrency

maka pola migration yang paling cocok adalah:

```text id="ep0vdj"
- 1 binary khusus migrate
- 1 folder migration per write database target
- versioning terpisah per target
- API tidak mengurus migration saat runtime normal
- read replica tidak punya migration runner
```

Kalau Anda mau, saya bisa lanjutkan dengan **template implementasi `cmd/migrate/main.go`** untuk project Anda, lengkap dengan:

* support `mysql_main`
* support `postgres_reporting`
* command `up`, `down`, `version`
* struktur folder migration yang langsung cocok dengan tree Anda sekarang.
