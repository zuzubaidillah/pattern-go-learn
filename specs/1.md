For your case, the best fit is:

**Modular monolith + Clean Architecture + explicit infrastructure adapters**

Why this is the right choice:

* **Modular monolith** keeps deployment and local development simple, while still giving you clear feature boundaries like `user`, `auth`, `exam`, `billing`, and `notification`. It is easier to maintain than jumping to microservices too early.
* **Gin** is a strong choice for high-throughput APIs and supports middleware, route grouping, and clean request handling patterns well. ([Gin Web Framework][1])
* **`database/sql`-style connection management** works well for multiple DB pools, read/write separation, and transaction control. Context-aware queries and `BeginTx` are important for production reliability. ([Go.dev][2])
* **Redis via go-redis** is a practical fit for cache, queue, token/session helpers, and rate limiting. ([Redis][3])
* **Structured logging with Zap** is a good production default because it is fast and made for hot paths. ([GitHub][4])

## 1. Recommended architecture pattern

Use this pattern:

**Transport → Use Case / Service → Repository → Infrastructure**

With module boundaries like:

* `internal/modules/user`
* `internal/modules/auth`
* `internal/modules/order`

Inside each module, separate:

* **transport/http**: Gin handler, request/response DTO, route registration
* **service**: business rules, orchestration, transactions
* **repository**: DB access interface + SQL implementation
* **domain**: entity, contracts, errors

That gives you:

* feature-based ownership
* low coupling
* easier testing
* clean migration later if one module must become a service

### Why not classic layered by technical type only?

A global structure like `handlers/`, `services/`, `repositories/` becomes hard to maintain at scale because all features mix together.

### Why not full DDD with too many abstractions?

You asked for practical implementation. For most REST APIs, a **light clean architecture** is better than over-engineered DDD.

## 3. Dependency flow and boundaries

Use this rule:

**outer layer may depend on inner layer, inner layer must not depend on outer layer**

### Allowed dependencies

* `transport/http` → depends on `service interface`
* `service` → depends on `repository interface`
* `repository/sql` → depends on DB adapter
* `infra` → concrete implementations only

### Not allowed

* service importing Gin
* domain importing SQL driver
* repository returning Gin responses
* handler directly writing SQL

### Clean dependency flow

```text
Gin Handler
   ↓
Service / Use Case
   ↓
Repository Interface
   ↓
Repository Implementation
   ↓
MySQL / PostgreSQL / Redis
```

### Important boundary rule

Define interfaces close to where they are used.

Example:

* `UserRepository` interface belongs in `modules/user/domain`
* concrete SQL repo belongs in `modules/user/repository`

That keeps the business layer in control.

### Connection builder

```go
// File: internal/infra/db/mysql.go
package db

import (
	"database/sql"
	"time"

	_ "github.com/go-sql-driver/mysql"
	"yourapp/internal/config"
)

func NewMySQL(conn config.DatabaseConn) (*sql.DB, error) {
	db, err := sql.Open("mysql", conn.DSN)
	if err != nil {
		return nil, err
	}

	db.SetMaxOpenConns(conn.MaxOpenConns)
	db.SetMaxIdleConns(conn.MaxIdleConns)
	db.SetConnMaxLifetime(conn.ConnMaxLifetime)
	db.SetConnMaxIdleTime(conn.ConnMaxIdleTime)

	if err := db.Ping(); err != nil {
		return nil, err
	}
	return db, nil
}

// reasonable example defaults:
// MaxOpenConns: 50-100 per instance
// MaxIdleConns: 10-30
// ConnMaxLifetime: 30m-1h
var _ = time.Second
```

### Practical rule for read/write separation

In repository layer:

* `Create`, `Update`, `Delete` → write DB
* `Get`, `List` → read DB
* transaction flows → always use write DB

That keeps behavior predictable.

### Transaction helper

Because `database/sql` supports `BeginTx` with context and auto rollback on context cancellation, use that in service methods that need consistency. ([Go.dev][2])

```go
// File: internal/infra/db/tx.go
package db

import (
	"context"
	"database/sql"
)

func WithTransaction(ctx context.Context, db *sql.DB, fn func(tx *sql.Tx) error) error {
	tx, err := db.BeginTx(ctx, nil)
	if err != nil {
		return err
	}

	defer func() {
		_ = tx.Rollback()
	}()

	if err := fn(tx); err != nil {
		return err
	}

	return tx.Commit()
}
```

## 5. Redis integration strategy

Use Redis for 3 separate responsibilities:

### A. Cache

Good for:

* user profile
* permission snapshot
* config lookup
* hot list endpoints

Use **cache-aside**:

1. handler/service asks cache
2. if miss, load from DB
3. save to Redis with TTL
4. return result

### B. Rate limit

Use Redis-based middleware because you want horizontal scaling and shared rate-limit state across instances.

### C. Queue

For lightweight jobs:

* send email
* audit logging
* webhook retry
* async post-processing

If queueing becomes mission-critical, you may later move to dedicated queue tech, but Redis is fine for many modular monoliths.

Redis production usage with the Go client is a standard and well-supported path. ([Redis][3])

### Redis client

```go
// File: internal/infra/redis/client.go
package redisx

import (
	"context"

	"github.com/redis/go-redis/v9"
	"yourapp/internal/config"
)

func NewClient(cfg config.RedisConfig) (*redis.Client, error) {
	rdb := redis.NewClient(&redis.Options{
		Addr:     cfg.Addr,
		Password: cfg.Password,
		DB:       cfg.DB,
	})

	if err := rdb.Ping(context.Background()).Err(); err != nil {
		return nil, err
	}

	return rdb, nil
}
```

### Cache wrapper

```go
// File: internal/infra/redis/cache.go
package redisx

import (
	"context"
	"encoding/json"
	"time"

	"github.com/redis/go-redis/v9"
)

type Cache struct {
	client *redis.Client
}

func NewCache(client *redis.Client) *Cache {
	return &Cache{client: client}
}

func (c *Cache) SetJSON(ctx context.Context, key string, value any, ttl time.Duration) error {
	b, err := json.Marshal(value)
	if err != nil {
		return err
	}
	return c.client.Set(ctx, key, b, ttl).Err()
}

func (c *Cache) GetJSON(ctx context.Context, key string, dest any) (bool, error) {
	val, err := c.client.Get(ctx, key).Result()
	if err == redis.Nil {
		return false, nil
	}
	if err != nil {
		return false, err
	}
	if err := json.Unmarshal([]byte(val), dest); err != nil {
		return false, err
	}
	return true, nil
}

func (c *Cache) Delete(ctx context.Context, keys ...string) error {
	return c.client.Del(ctx, keys...).Err()
}
```

### Rate limit middleware idea

Use key like:

`rl:{route}:{userID}` or `rl:{ip}:{route}`

For example:

* login: per IP
* answer submit: per user + exam id
* resend OTP: per phone/email

### Queue strategy

Keep producer/consumer separate:

* API writes job payload into Redis list/stream
* Worker consumes independently

That preserves stateless API nodes.

## Repository implementation

```go
// File: internal/modules/user/repository/sql_repository.go
package repository

import (
	"context"
	"database/sql"
	"errors"

	"yourapp/internal/modules/user/domain"
)

type SQLRepository struct {
	writeDB *sql.DB
	readDB  *sql.DB
}

func NewSQLRepository(writeDB, readDB *sql.DB) *SQLRepository {
	return &SQLRepository{
		writeDB: writeDB,
		readDB:  readDB,
	}
}

func (r *SQLRepository) FindByID(ctx context.Context, id int64) (*domain.User, error) {
	const q = `
		SELECT id, name, email, status, created_at, updated_at
		FROM users
		WHERE id = ? LIMIT 1
	`

	var user domain.User
	err := r.readDB.QueryRowContext(ctx, q, id).Scan(
		&user.ID,
		&user.Name,
		&user.Email,
		&user.Status,
		&user.CreatedAt,
		&user.UpdatedAt,
	)
	if err != nil {
		if errors.Is(err, sql.ErrNoRows) {
			return nil, nil
		}
		return nil, err
	}

	return &user, nil
}

func (r *SQLRepository) FindByEmail(ctx context.Context, email string) (*domain.User, error) {
	const q = `
		SELECT id, name, email, status, created_at, updated_at
		FROM users
		WHERE email = ? LIMIT 1
	`

	var user domain.User
	err := r.readDB.QueryRowContext(ctx, q, email).Scan(
		&user.ID,
		&user.Name,
		&user.Email,
		&user.Status,
		&user.CreatedAt,
		&user.UpdatedAt,
	)
	if err != nil {
		if errors.Is(err, sql.ErrNoRows) {
			return nil, nil
		}
		return nil, err
	}

	return &user, nil
}

func (r *SQLRepository) Create(ctx context.Context, user *domain.User) error {
	const q = `
		INSERT INTO users (name, email, status, created_at, updated_at)
		VALUES (?, ?, ?, NOW(), NOW())
	`

	res, err := r.writeDB.ExecContext(ctx, q, user.Name, user.Email, user.Status)
	if err != nil {
		return err
	}

	id, err := res.LastInsertId()
	if err != nil {
		return err
	}

	user.ID = id
	return nil
}

func (r *SQLRepository) Update(ctx context.Context, user *domain.User) error {
	const q = `
		UPDATE users
		SET name = ?, status = ?, updated_at = NOW()
		WHERE id = ?
	`

	_, err := r.writeDB.ExecContext(ctx, q, user.Name, user.Status, user.ID)
	return err
}
```

## Service

```go
// File: internal/modules/user/service/service.go
package service

import (
	"context"
	"errors"
	"fmt"
	"time"

	"yourapp/internal/modules/user/domain"
	"yourapp/internal/infra/redis"
)

type Service struct {
	repo  domain.Repository
	cache *redisx.Cache
}

func New(repo domain.Repository, cache *redisx.Cache) *Service {
	return &Service{
		repo:  repo,
		cache: cache,
	}
}

func (s *Service) GetByID(ctx context.Context, id int64) (*domain.User, error) {
	cacheKey := fmt.Sprintf("user:%d", id)

	var cached domain.User
	found, err := s.cache.GetJSON(ctx, cacheKey, &cached)
	if err == nil && found {
		return &cached, nil
	}

	user, err := s.repo.FindByID(ctx, id)
	if err != nil {
		return nil, err
	}
	if user == nil {
		return nil, nil
	}

	_ = s.cache.SetJSON(ctx, cacheKey, user, 5*time.Minute)
	return user, nil
}

func (s *Service) Create(ctx context.Context, req domain.CreateUserInput) (*domain.User, error) {
	existing, err := s.repo.FindByEmail(ctx, req.Email)
	if err != nil {
		return nil, err
	}
	if existing != nil {
		return nil, errors.New("email already exists")
	}

	user := &domain.User{
		Name:   req.Name,
		Email:  req.Email,
		Status: req.Status,
	}

	if err := s.repo.Create(ctx, user); err != nil {
		return nil, err
	}

	return user, nil
}

func (s *Service) Update(ctx context.Context, req domain.UpdateUserInput) (*domain.User, error) {
	user, err := s.repo.FindByID(ctx, req.ID)
	if err != nil {
		return nil, err
	}
	if user == nil {
		return nil, nil
	}

	user.Name = req.Name
	user.Status = req.Status

	if err := s.repo.Update(ctx, user); err != nil {
		return nil, err
	}

	_ = s.cache.Delete(ctx, fmt.Sprintf("user:%d", user.ID))
	return user, nil
}
```

## HTTP handler

```go
// File: internal/modules/user/transport/http/handler.go
package http

import (
	"net/http"
	"strconv"

	"github.com/gin-gonic/gin"
	"yourapp/internal/modules/user/domain"
)

type Handler struct {
	service domain.Service
}

func NewHandler(service domain.Service) *Handler {
	return &Handler{service: service}
}

func (h *Handler) GetByID(c *gin.Context) {
	id, err := strconv.ParseInt(c.Param("id"), 10, 64)
	if err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"message": "invalid user id"})
		return
	}

	user, err := h.service.GetByID(c.Request.Context(), id)
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"message": "internal server error"})
		return
	}
	if user == nil {
		c.JSON(http.StatusNotFound, gin.H{"message": "user not found"})
		return
	}

	c.JSON(http.StatusOK, UserResponse{
		ID:     user.ID,
		Name:   user.Name,
		Email:  user.Email,
		Status: user.Status,
	})
}

func (h *Handler) Create(c *gin.Context) {
	var req CreateUserRequest
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"message": "invalid request", "error": err.Error()})
		return
	}

	user, err := h.service.Create(c.Request.Context(), domain.CreateUserInput{
		Name:   req.Name,
		Email:  req.Email,
		Status: req.Status,
	})
	if err != nil {
		if err.Error() == "email already exists" {
			c.JSON(http.StatusConflict, gin.H{"message": err.Error()})
			return
		}
		c.JSON(http.StatusInternalServerError, gin.H{"message": "internal server error"})
		return
	}

	c.JSON(http.StatusCreated, UserResponse{
		ID:     user.ID,
		Name:   user.Name,
		Email:  user.Email,
		Status: user.Status,
	})
}

func (h *Handler) Update(c *gin.Context) {
	id, err := strconv.ParseInt(c.Param("id"), 10, 64)
	if err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"message": "invalid user id"})
		return
	}

	var req UpdateUserRequest
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"message": "invalid request", "error": err.Error()})
		return
	}

	user, err := h.service.Update(c.Request.Context(), domain.UpdateUserInput{
		ID:     id,
		Name:   req.Name,
		Status: req.Status,
	})
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"message": "internal server error"})
		return
	}
	if user == nil {
		c.JSON(http.StatusNotFound, gin.H{"message": "user not found"})
		return
	}

	c.JSON(http.StatusOK, UserResponse{
		ID:     user.ID,
		Name:   user.Name,
		Email:  user.Email,
		Status: user.Status,
	})
}
```

## 7. Best practices

## A. Dependency injection

Do not use global variables for DB, Redis, logger.

Use constructor injection:

```go
repo := repository.NewSQLRepository(dbm.MySQLWrite, dbm.MySQLRead)
svc := service.New(repo, cache)
h := userhttp.NewHandler(svc)
```

That makes testing and replacement easy.

### Good pattern

* `main.go` bootstraps infra
* `module.go` assembles feature dependencies
* handlers/services/repositories receive only what they need

---

## B. Config management

Use:

* environment variables for secrets
* YAML only for non-secret defaults if you want
* a single parsed config struct
* fail fast on startup

Example bootstrapping flow:

```go
cfg := MustLoadConfig()
logger := NewLogger(cfg.Log)
dbm := MustNewDBManager(cfg.DB)
redisClient := MustNewRedis(cfg.Redis)
cache := redisx.NewCache(redisClient)
```

Keep config centralized.

---

## C. Logging

Use structured logging, not `fmt.Println`.

Zap is a strong production choice for performance-sensitive paths. ([GitHub][4])

Example:

```go
// File: internal/infra/logger/zap.go
package logger

import "go.uber.org/zap"

func New(env string) (*zap.Logger, error) {
	if env == "production" {
		return zap.NewProduction()
	}
	return zap.NewDevelopment()
}
```

Log with fields:

```go
logger.Info("user created",
	zap.Int64("user_id", user.ID),
	zap.String("email", user.Email),
)
```

Always include:

* request_id
* user_id if available
* module
* latency
* status_code
* error_code

## D. Error handling

Use one application error format.

Example:

```go
// File: internal/pkg/errors/app_error.go
package errors

type AppError struct {
	Code    string
	Message string
	Err     error
}

func (e *AppError) Error() string {
	return e.Message
}
```

Then map error → HTTP response centrally.

Example mapping:

* validation error → 400
* unauthorized → 401
* forbidden → 403
* not found → 404
* conflict → 409
* internal → 500

Do not leak raw SQL errors to clients.

## 8. High concurrency and horizontal scaling

For **3000+ users**, these are the practical rules:

### App layer

* keep API nodes stateless
* no in-memory session state
* use Redis/shared store for distributed rate limit or ephemeral state
* set server timeouts
* use request context everywhere

### DB layer

* tune connection pools per instance carefully
* do not set huge `MaxOpenConns` blindly
* add indexes based on actual query patterns
* split reads to replica if needed
* avoid N+1 queries

### Redis layer

* use for hot-path cache and throttling
* set TTLs explicitly
* namespace keys clearly:

  * `user:123`
  * `ratelimit:login:1.2.3.4`

### Deployment

* multiple API instances behind LB
* graceful shutdown
* readiness/liveness checks
* worker as separate process

## Final design rule

* **handler** = HTTP translation only
* **service** = business decision only
* **repository** = persistence only
* **infra** = connection/client only
